# ETSI EN 303 645 — Evidence walk-through

**Standard:** ETSI EN 303 645 V2.1.1 (2020-06) — *Cyber Security for Consumer
Internet of Things: Baseline Requirements*.
**Status of this document:** Internal evidence map for the M6 W2
walk-through. Each row of the table cites the file path or commit
that satisfies (or partially satisfies) the provision. Treated as
audit-ready notes, not as a formal cert claim — formal certification
is a post-M6 commercial track per `docs/M5-PLAN.md`.
**Last updated:** M6 W2 (this commit).

The standard has 13 numbered provisions. Each row below pairs the
provision text (paraphrased — read the standard for the canonical
language) with our evidence and a maturity grade:

* **C** — *compliant*: evidence in tree, tests covering the surface.
* **P** — *partial*: evidence exists, gaps documented in the row's
  notes column.
* **N/A** — provision doesn't apply to our deployment shape (rare).
* **O** — *open*: evidence to be produced before `v1.0.0`.

| § | Provision | Status | Evidence | Notes |
|---|-----------|--------|----------|-------|
| 5.1 | No universal default passwords | **C** | Per-plugin NATS user JWTs + nkeys minted at install time (`crates/iot-cli/src/plugin.rs:520` `generate_nats_identity`); broker decentralised auth via operator-signed account JWTs (M5a W1, ADR-0011 retired). Mosquitto ACL regenerated per plugin manifest (`crates/iot-cli/src/mosquitto.rs`). No shared default credentials anywhere. | The original M1 dev mode used a `no_auth_user` placeholder; that retired in M5a W1. |
| 5.2 | Implement a means to manage reports of vulnerabilities | **C** | `SECURITY.md` at repo root (90-day embargo, scope, contact). RFC 9116 `security.txt` at `panel/public/.well-known/security.txt` + a gateway endpoint at `/.well-known/security.txt` (`crates/iot-gateway/src/handlers.rs::well_known_security_txt`). GitHub Security Advisories private-report path live. | Email + PGP key are placeholders pre-`v1.0.0`. M6 W4 finalises both. |
| 5.3 | Keep software updated | **C** | Cosign keyless sign-blob + Rekor entry on every release artifact (`.github/workflows/ci.yml` `sign` job). SBOM CycloneDX per crate (`sbom` job). Plugin install enforces SBOM CVE gate (`iotctl plugin install` rejects high/critical advisories without `--allow-vulnerabilities`; `crates/iot-cli/src/plugin.rs:192` `enforce_sbom_policy`). Dependabot enabled + advisory-DB (`deny.toml [advisories]`). | Update channel for the hub binary itself is the GH Releases page; auto-update is post-M6 (TUF root rotation rehearsed in M6 W4). |
| 5.4 | Securely store sensitive security parameters | **C** | POSIX `chmod 0600` + Windows ACL via `icacls` on every secret-bearing file the install path writes (`crates/iot-cli/src/secfile.rs`). The audit Bucket 1 fix (commit `55753b0`) closed the previous Windows no-op stub. Plugin nkeys, NATS creds, operator + account seeds all gated. | No HSM integration; secrets sit on local disk under restrictive perms. Acceptable for hub-class hardware. |
| 5.5 | Communicate securely | **C** | mTLS everywhere on the bus (`crates/iot-bus/src/lib.rs::Bus::connect` requires `add_root_certificates` + `add_client_certificate`). MQTT broker uses rustls (`rumqttc` `use-rustls` feature, ADR-0006). Gateway HTTP+WS behind Envoy in prod. Cert-rotation behaviour documented in `docs/security/cert-rotation-test.md` with operator runbook + threat model + unit-pin (M6 W2.5). | Live cert-reload (no process restart) is documented as a post-M6 hardening item, not a current gap; restart-rotation is the supported path. |
| 5.6 | Minimize exposed attack surfaces | **C** | WASM plugin capability model: every host call gates on the plugin's manifest-declared ACL (`crates/iot-plugin-host/src/capabilities.rs`). Audit Bucket 1 hardened the `net.outbound` URL match (case canonicalisation, scheme/userinfo/path-prefix). Closed-domain voice grammar rejects free-form text (`crates/iot-voice/src/intent.rs::RuleIntentParser`); LLM-fallback NLU is M5b W4+ scope and stays a separate path. Bus subject taxonomy enforced at publish time (manifest `bus.publish` allow-list). | NATS subjects under `_INBOX.>` and `$JS.>` are NATS-internal and not gated by plugin ACL; they're broker-internal traffic. |
| 5.7 | Ensure software integrity | **C** | Cosign blob signature + Rekor lookup on `iotctl plugin install` (`crates/iot-cli/src/plugin.rs:495` `verify_signature`). Rekor entry recorded at release time. SLSA L3 provenance attestation hard-gated post-M6 W1 (`/.github/workflows/ci.yml::sign job::SLSA provenance`). Reproducibility byte-match via diffoscope on every tag (`reproducibility` job; strict via `bash -eo pipefail`). | The plugin install path exposes `--allow-unsigned` for dev; prod-build gating per ADR-0006 keeps it inaccessible. |
| 5.8 | Ensure that personal data is secure | **P** | History capture is opt-in (`IOT_TIMESCALE_URL` unset = no capture; `crates/iot-history/src/lib.rs::from_env`). When enabled, rows are stored in TimescaleDB which the operator runs locally (no cloud dependency). Deletion: `iotctl history prune --device-id <ulid>` and `--before <rfc3339>` (M6 W2 deliverable). TLS-at-rest runbook documented in `docs/security/asvs-l2.md` § "Operator runbook for TLS-at-rest" (M6 W3). | **Partial:** the project ships unencrypted-by-default; operator action required to enable LUKS-mount or `pgcrypto`. A turn-key encrypted-history bootstrap is a post-M6 deliverable (M7+). |
| 5.9 | Make systems resilient to outages | **C** | Plugin supervisor restarts on crash with exponential back-off + DLQ after 5 successive crashes (`crates/iot-plugin-host/src/supervisor.rs`). Rule engine action failures publish to `sys.automation.dlq` instead of dropping silently (`crates/iot-automation/src/engine.rs:70`). Bus reconnect on transient NATS failures via `async-nats` defaults. Audit chain doesn't lose entries on broker disconnect (M3 W1.4 hash-chain durable through reconnects). | Hub-host crash recovery is systemd's job in prod; we don't ship a custom watchdog. |
| 5.10 | Examine system telemetry data | **C** | Append-only audit log with JCS-canonical (RFC 8785) hash chain (`crates/iot-audit/src/lib.rs`). Every plugin host-call records an entry (`audit.plugin.bus_publish`, `audit.capability.denied`, etc). Rule firings audit-logged (`automation.rule_fired`). Action failures audit-logged + DLQ-published. Tamper-evident via the chain's prev-hash pointer. | Operators can `nats sub sys.audit.>` to live-tail; offline analysis via the on-disk JCS journal. |
| 5.11 | Make it easy for users to delete user data | **C** | `iotctl history prune --device-id <ulid>` deletes every row for one device (M6 W2 deliverable, this commit). `--before <rfc3339>` deletes by time cutoff. Confirmation prompt unless `--yes`. The crate-level prune API (`crates/iot-history/src/lib.rs::prune_for_device`) takes optional time cutoff so combined filters (this device, before this date) work too. | Refuses the no-arg "delete everything" path by design — the SQL `DROP TABLE` escape hatch is documented but kept out of the CLI. |
| 5.12 | Make installation and maintenance of devices easy | **C** | `just dev` brings up the entire compose stack (NATS, Mosquitto, registry, gateway, panel) from a fresh checkout. `iotctl plugin install <path>` is the entire plugin onboarding ritual. `iot-voice send "<text>"` smoke-tests the voice loop end-to-end without hardware. `docs/LOCAL-CI.md` walks operator iteration without paying GH Actions runner time. | Hub-class deployment (Pi 5) is single-binary `iot-host` + the WASM plugin dir; no service-mesh-grade orchestration. |
| 5.13 | Validate input data | **C** | Manifest schema validated via JSON-schema before the plugin install accepts (`schemas/plugin-manifest.schema.json`). Capability ACLs gate every host call: subject-prefix match for bus.publish, path-prefix for net.outbound (Bucket 1 hardened canonicalisation), forbidden-header deny-list. CEL expression source is capped at 64 KB pre-parse + 200 ms eval timeout (`crates/iot-automation/src/expr.rs::MAX_EXPR_SOURCE_BYTES`, `EVAL_TIMEOUT`; audit Bucket 2 H2). Intent grammar rejects unrecognised phrases (no LLM fallback in default path). MQTT topic-filter validation in plugin manifests. | Generic untrusted-payload validation lives at the bus subject's protobuf schema layer; our schemas use `prost` so wire-level validation is automatic. |

## Coverage summary

* **Compliant:** 12 / 13 provisions (5.1, 5.2, 5.3, 5.4, 5.5, 5.6, 5.7, 5.9, 5.10, 5.11, 5.12, 5.13)
* **Partial:** 1 / 13 (5.8 TLS-at-rest is operator-action; runbook documented in asvs-l2.md)
* **Open:** 0
* **N/A:** 0

The remaining partial has explicit M6 follow-up + a documented operator path; not a structural blocker.

## Walk-through methodology

This document was produced by:

1. Mapping each ETSI provision to the project's nearest existing
   feature surface (M1-M6 work).
2. Citing file paths + commits where the provision is enforced.
3. Grading evidence as C/P/O against an "audit-ready" bar — i.e.
   could an external assessor follow the citation, find the code,
   and verify the assertion? If the answer's "yes," it's at least
   P; if the test surface also exists, it's C.
4. Flagging gaps as M6.x or post-M6 follow-ups, not silently
   accepting them.

## Out-of-scope provisions

ETSI EN 303 645 also references Annex A (data-protection
provisions, mostly GDPR-aligned) and Annex B (implementation
guidance per device class). Both are advisory; this evidence map
covers the normative provisions in §5 only. GDPR/CCPA-style
data-subject rights ride on §5.11 + the operator's deployment
choices (data-minimisation defaults, retention windows) and don't
need their own evidence rows here.
